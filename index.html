QUIRKI

An experimental programming language. It is fast, memory-safe and data-oriented. It's very opinionated, and very quirki.

Functions & Ownership

Functions do not return values. They mutate data instead, keeping ownership obvious and reasoning simple. 


// Instead of using keywords, Quirki uses positional syntax.
// Mutating parameters come first, followed by function 
// and readonly parameters.

new hero // Data is zero initialized on creation.

hero TakeDamage(i_amount)
hero Move(f3_dir)
      
// Can also be written with pipeline syntax:
hero
| TakeDamage(i_amount)
| Move(f3_dir)

// You can also take ownership with keyword: take
hero GrabItem(take item)

// And have multiple mutable parameters:
enemy, ammo Shoot(weapon)

Variables, Names & Aliases

The language is strongly typed, but it does so by inferring types from variable names using type names as prefixes. In Quirki you are encouraged to use the type name as the variable name. 


// Other Similar Language
fn PushDamage(hero: mut &Hero, transform:&Transform, weapon:Weapon, amount: Int)
  ...

// Quirki:
=== hero <- PushDamage(weapon, transform, i_amount)
  ...

=== Hero:
   entity 
   weapon  
   i_health

// Parametric polymorphism
var item_list // list of items.

// For complex types, prefix with _
var _adresses : person_f2_map 

// Type aliases
=== Alias Adresses : person_f2_map 
=== Alias is     : b
=== Alias has    : b

// It's obvious when types mismatch
i_health -= f_damage

Enums & Errors

Enums are used heavily. Cases may carry values and can model failures.


=== Enum FireWeapon // simple enum
    success
    noAmmo
    jammed
    overheated

=== Enum AIState // enum with data
    Idle
    Patrol(f3_dir)
    Attack(entity)
    Flee(f3_dir, f_speed)

// Error-capable enums enable ! unwrapping
=== Enum Optional
    !None
    Some(T)

=== Enum LoadResult
    !Error
    Success(asset)

// Rule: enums with error variants must use one payload type for all successes.

=== LoadResult <- LoadAsset(path)
    ! asset LoadAsset(path)      // default failure is !Error
    ! asset LoadAsset(path) : AboveResultEnum.AssetFailLoad

// Anonymous enums while declaring functions
=== ! or Success(asset) <- LoadAsset(path)

// Error handling in use
? LoadTexture(path)
| Success(asset):   loadedAssets(take asset)
| NotFound:         Log("Texture not found")
| PermissionDenied: Log("Permission denied")

// Unwrap success and propagate failure automatically.
! texture LoadTexture(path)

Branching & Pattern Matching

There is only one control-flow construct: ?. It pattern-matches on data and is exhaustive unless else is used. No if/switch/try/catch. 


? hero.ai_state
| Idle: hero Rest()
| Patrol(dir): hero Move(dir)
| else: pass // use else to catch-all
    
// Optionals look the same
? hero.target
| some(entity): hero Shoot(entity)
| none: hero Search()

// # introduces a temporary value to match on
? # GetTotalAmmo(hero)
| # > 0: hero Shoot()
| else: hero Reload()

// And since optionals are just enums with a failure variant:
! enemy hero.target

// Boolean is an enum with sugar so you don't need "| true:"
? _ CheckAmmo(hero)
    hero Shoot()
| else:
    hero Reload()

! asset LoadAsset(path)

Initialization

Marking a function with init makes it a constructor. The compiler enforces that all fields are set and none are read before assignment. 


=== hero <- Init (data)
    hero.entity = data.entity
    hero.weapon = data.weapon
    hero.i_health = data.health

=== hero <- Init fromHealth(data, i_health)
    hero.entity = data.entity
    hero.weapon = data.weapon
    hero.i_health = i_health

=== Example
    new weapon
    use(weapon) // Compiler Error: not initialized

    //-----------------------------------------

    new weapon init(path)
    use(weapon)  // OK

    //-----------------------------------------

    new weapon

    ? hero.class
    | Warrior: 
        weapon Init(sword_data)
    | Archer:
        weapon Init(bow_data)

    hero Equip(weapon) // OK
